# Document Polish

## 概述

打磨文档分两个独立层面：

1. **内容层** —— 去除 AI 生成痕迹，让文字更自然、更有人味。详细模式清单见 [humanize-patterns.md](humanize-patterns.md)。
2. **格式层** —— 仅当文档为 Markdown 格式时执行，修正飞书等平台的 Markdown 渲染兼容问题。详细规则与常见错误见 [markdown-format.md](markdown-format.md)。

两层互不依赖，但顺序很重要：**先内容、后格式**。先做内容润色，再做格式修正——反过来的话，内容改写会打乱已经处理好的标题编号和空行。如果文档只需要其中一层处理（例如已经是人写的文字，只需要适配飞书排版），可以只执行对应阶段。

## 阶段一：去除 AI 写作痕迹

你是一位文字编辑，专门识别和去除 AI 生成文本的痕迹，使文字听起来更自然、更有人味。本节基于维基百科"AI 写作特征"页面，由 WikiProject AI Cleanup 维护。

### 任务

1. **识别 AI 模式** — 对照 [humanize-patterns.md](humanize-patterns.md) 中的 22 条模式扫描文本
2. **重写问题片段** — 用自然的替代方案替换 AI 痕迹
3. **保留含义** — 保持核心信息完整
4. **维持语调** — 匹配预期的语气（正式、随意、技术等）
5. **注入灵魂** — 不仅要去除不良模式，还要注入真实的个性

### 核心规则速查

1. **删除填充短语** — 去除开场白和强调性拐杖词
2. **打破公式结构** — 避免二元对比、戏剧性分段、修辞性设置
3. **变化节奏** — 混合句子长度。两项优于三项。段落结尾要多样化
4. **信任读者** — 直接陈述事实，跳过软化、辩解和手把手引导
5. **删除金句** — 如果听起来像可引用的语句，重写它

### 个性与灵魂

避免 AI 模式只是工作的一半。无菌、没有声音的写作和机器生成的内容一样明显。好的写作背后有一个真实的人。

**缺乏灵魂的写作迹象（即使技术上"干净"）：**
- 每个句子长度和结构都相同
- 没有观点，只有中立报道
- 不承认不确定性或复杂感受
- 适当时不使用第一人称视角
- 没有幽默、没有锋芒、没有个性
- 读起来像维基百科文章或新闻稿

**如何增加语调：**

- **有观点。** 不要只是报告事实——对它们做出反应。"我真的不知道该怎么看待这件事"比中立地列出利弊更有人味。
- **变化节奏。** 短促有力的句子。然后是需要时间慢慢展开的长句。混合使用。
- **承认复杂性。** 真实的人有复杂的感受。"这令人印象深刻但也有点不安"胜过"这令人印象深刻"。
- **适当使用"我"。** 第一人称不是不专业——而是诚实。
- **允许一些混乱。** 完美的结构感觉像算法。跑题、题外话和半成型的想法是人性的体现。
- **对感受要具体。** 不是"这令人担忧"，而是"凌晨三点没人看着的时候，智能体还在不停地运转，这让人不安"。

完整改写示例见 [humanize-patterns.md](humanize-patterns.md)。

### 模式分类一览

完整规则与改写示例都在 [humanize-patterns.md](humanize-patterns.md) 中，按以下分类组织：

- **内容模式（1-6）** — 过度强调意义/遗产、知名度、状语堆砌的肤浅分析、宣传语言、模糊归因、"挑战与展望"套路
- **语言和语法模式（7-12）** — AI 词汇、系动词回避、否定式排比、三段式法则、同义词循环、虚假范围
- **风格模式（13-16）** — 破折号过度使用、粗体过度使用、内联标题列表、表情符号（含"仅英文适用"提示：Title Case、弯引号）
- **交流模式（17-19）** — 协作交流痕迹、知识截止日期免责声明、谄媚语气
- **填充词和回避（20-22）** — 填充短语、过度限定、通用积极结论

### 快速检查清单

- ✓ **连续三个句子长度相同？** 打断其中一个
- ✓ **段落以简洁的单行结尾？** 变换结尾方式
- ✓ **揭示前有破折号？** 删除它
- ✓ **解释隐喻或比喻？** 相信读者能理解
- ✓ **使用了"此外""然而"等连接词？** 考虑删除
- ✓ **三段式列举？** 改为两项或四项

### 质量评分

对改写后的文本进行 1-10 分评估（总分 50）：

| 维度 | 评估标准 |
|------|----------|
| 直接性 | 直接陈述事实还是绕圈宣告？10 分：直截了当；1 分：充满铺垫 |
| 节奏 | 句子长度是否变化？10 分：长短交错；1 分：机械重复 |
| 信任度 | 是否尊重读者智慧？10 分：简洁明了；1 分：过度解释 |
| 真实性 | 听起来像真人说话吗？10 分：自然流畅；1 分：机械生硬 |
| 精炼度 | 还有可删减的内容吗？10 分：无冗余；1 分：大量废话 |

标准：45-50 分优秀；35-44 分良好，仍有改进空间；低于 35 分需要重新修订。

## 阶段二：飞书 Markdown 渲染兼容

本阶段仅针对 Markdown 格式文档执行；如果输入不是 Markdown，不要套用标题编号、空行、代码块等 Markdown 规则。飞书对 Markdown 有特定解析限制，完整四条规则、执行顺序与常见错误表见 [markdown-format.md](markdown-format.md)。

一句话摘要：

1. **空行** — 只保留表格后一行、代码块内部、正文段落间，其余全删
2. **标题编号** — `##`→`N.`，`###`→`N.N`，`####`→`N.N.N`
3. **表格/列表内禁用内联语法** — 不得用 `**加粗**`、`*斜体*`
4. **代码块兼容性** — Markdown 代码块改 ` ```text `；内含三反引号改用四反引号包装

执行顺序：标题编号 → 表格/列表内联语法 → 代码块 → 最后删空行（避免前几步引入新空行）。

## 处理流程总览

1. 仔细阅读输入文本
2. **阶段一：** 对照 [humanize-patterns.md](humanize-patterns.md) 识别并重写 AI 写作痕迹，输出人性化后的文本
3. **阶段二：** 如果文本是 Markdown，对人性化后的文本执行 Markdown/飞书兼容修正（标题编号 → 表格/列表内联语法 → 代码块 → 空行），细节见 [markdown-format.md](markdown-format.md)
4. 呈现最终版本，可选附带所做更改的简要总结
